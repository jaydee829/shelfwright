# Runbook: GCP Walking Skeleton (Lift 0)

One-time provisioning + restore for the production environment. Scripts live in
`infra/`; run them FROM THE WSL CLONE (it holds the `data/backups/` dump, which is
gitignored — a fresh clone does NOT have it), in numbered order. Total cost when
done: ~$12–16/month (Cloud SQL is the floor; everything else is cents).

> **One-shot semantics:** these scripts are NOT idempotent. Re-running a
> partially-completed step fails on ALREADY_EXISTS by design (fail-loud beats
> silent-skip). To recover: delete the offending resource, or skip past the
> completed commands and run the rest by hand.

---

## 0. Prerequisites (one-time tooling)

Install the Google Cloud CLI in WSL following the Debian/Ubuntu apt repository
instructions at <https://cloud.google.com/sdk/docs/install#deb>. The short version:

```bash
curl https://packages.cloud.google.com/apt/doc/apt-key.gpg | sudo gpg --yes --dearmor -o /usr/share/keyrings/cloud.google.gpg
echo "deb [signed-by=/usr/share/keyrings/cloud.google.gpg] https://packages.cloud.google.com/apt cloud-sdk main" \
  | sudo tee /etc/apt/sources.list.d/google-cloud-sdk.list
sudo apt update && sudo apt install -y google-cloud-cli
```

Authenticate and prime the local config (this opens a browser tab):

```bash
gcloud init        # walk the prompts; pick any existing project for now — 01-project.sh takes over
gcloud auth login  # if gcloud init didn't already authenticate
```

Find your billing account ID — you will need it for steps 1 and 10:

```bash
gcloud billing accounts list
export BILLING_ACCOUNT_ID=XXXXXX-XXXXXX-XXXXXX   # paste your value here
```

All scripts source `infra/00-config.sh` at startup. That file sets the project ID
(`agentic-librarian-prod`) and immediately runs `gcloud config set project` —
ensuring every subsequent command targets the right project even if your gcloud
config was last pointed elsewhere. It is safe to source before the project exists
(the `config set` command does not validate existence).

## 1. Project + APIs — `infra/01-project.sh`

**What it creates:** a GCP project named `agentic-librarian-prod`, linked to your
billing account, with all nine APIs the stack needs enabled: Cloud Run, Cloud SQL
Admin, Secret Manager, Artifact Registry, IAM Credentials, Cloud Resource Manager,
IAM, Billing Budgets, and Cloud Storage.

```bash
bash infra/01-project.sh
```

**PROJECT_ID collision:** GCP project IDs are globally unique across all customers.
If `agentic-librarian-prod` is already taken, override before running:

```bash
PROJECT_ID=agentic-librarian-prod-yourhandle bash infra/01-project.sh
```

**If billing link fails:** the project will have been created but will be in an
unbilled state — `services enable` will refuse to run. Fix billing in the GCP
console (Billing → Link a billing account → select the project), then re-run the
script. The `gcloud projects create` line will print a warning about the project
already existing and continue; that is expected.

**Cost:** $0. Creating a project and enabling APIs has no charge.

## 2. Cloud SQL — `infra/02-cloudsql.sh`

**What it creates:** a Cloud SQL instance called `librarian-sql` running Postgres 16
on a `db-f1-micro` shared-core machine (1 vCPU shared, 614 MB RAM) with a 10 GB SSD
in `us-central1`. Inside it, an empty database named `agentic_librarian`.

```bash
bash infra/02-cloudsql.sh
```

This step takes roughly 10 minutes — GCP is provisioning real hardware. Let it run.

At the end the script prints the **connection name** — a string of the form
`agentic-librarian-prod:us-central1:librarian-sql`. Copy it; you will need it in
step 5 when you set the `GCP_CLOUDSQL_CONNECTION` GitHub variable.

**Cost:** ~$12/month. This is the bill's floor — the instance runs 24/7 whether
or not the app is receiving traffic. Cloud Run scales to zero when idle, so SQL is
the only always-on cost.

---

## 3. DB user + secret — `infra/03-db-user-secret.sh`

**What it creates:** a Postgres user named `librarian` with a 48-character hex
password generated by `openssl rand -hex 24`. The password is URL-safe (hex only)
so it can sit inside a `postgresql://` connection string without escaping. The full
connection URL is immediately stored in Secret Manager under the name
`librarian-db-url` in this form:

```
postgresql://librarian:<hex-password>@/agentic_librarian?host=/cloudsql/<connection-name>
```

The `?host=/cloudsql/...` syntax is how SQLAlchemy reaches Cloud SQL via Unix socket
inside a Cloud Run container. Cloud Run injects the secret verbatim as `DATABASE_URL`
at runtime — the app never sees plain credentials.

```bash
bash infra/03-db-user-secret.sh
```

**The password exists only inside the secret.** It is never written to disk or shell
history beyond the process that ran `openssl rand`.

**Cost:** cents. Secret Manager charges ~$0.06/secret/month plus tiny access fees.

---

## 4. Registry + bucket — `infra/04-registry-bucket.sh`

**What it creates:** two storage resources.

1. An Artifact Registry Docker repository named `librarian` in `us-central1`. Every
   image the CI pipeline builds is tagged with the git commit SHA and pushed here
   (e.g., `us-central1-docker.pkg.dev/agentic-librarian-prod/librarian/librarian-api:abc1234`).
   Using the SHA as the tag makes every deployment traceable to an exact commit.

2. A Cloud Storage bucket named `gs://agentic-librarian-prod-backups` with
   uniform-bucket-level access. The restore script in step 6 uploads the pg_dump
   here so that Cloud SQL can pull it directly — Cloud SQL's import mechanism reads
   from GCS, not from your local machine.

```bash
bash infra/04-registry-bucket.sh
```

**Cost:** cents. Artifact Registry charges for storage (a few images cost pennies).
The bucket only holds the dump during the import and can be left in place cheaply.

## 5. IAM + WIF — `infra/05-iam-wif.sh`

**What it creates:** two service accounts, a Workload Identity Federation pool, and
all the bindings that let GitHub Actions deploy without a stored key.

**Service accounts (least-privilege split):**

- `librarian-api-runtime` — what the Cloud Run service runs as. It has exactly two
  permissions: read the `librarian-db-url` secret, and connect to Cloud SQL as a
  client. It cannot push images, deploy itself, or read arbitrary GCS data.
- `github-deployer` — what CI impersonates. It can push images to Artifact Registry,
  administer Cloud Run services (`roles/run.admin` — deploy is what we use it for),
  invoke (call) the service for smoke tests, and impersonate `librarian-api-runtime`
  (needed to deploy with that SA). It cannot read secrets or query the database.

**SA propagation gate:** GCP propagates new service accounts asynchronously. Binding
IAM policies to an SA that hasn't fully propagated yet returns a cryptic 400 error.
The script polls both SAs — up to 60 seconds, checking every 5 seconds — before
applying any bindings. You will see lines like `Waiting for ... to propagate (1/12)...`
if the SA isn't ready immediately; that is normal.

**Workload Identity Federation:** instead of a long-lived JSON key for CI, the
workflow exchanges a short-lived GitHub OIDC token for GCP credentials at runtime.
The pool is pinned to exactly `jaydee829/agentic_librarian` via an
`attribute.repository` condition — a fork or another repo cannot impersonate the
deployer SA. Nothing to rotate, nothing to leak.

```bash
bash infra/05-iam-wif.sh
```

At the end the script prints four values. Set each one as a **GitHub repo Variable**
(not a Secret) at:
**Settings → Secrets and variables → Actions → Variables tab → New repository variable**

| Variable name | Where the value comes from |
|---|---|
| `GCP_PROJECT_ID` | `agentic-librarian-prod` (or your override) |
| `GCP_WIF_PROVIDER` | printed by the script — starts with `projects/` |
| `GCP_DEPLOYER_SA` | printed by the script — ends with `.iam.gserviceaccount.com` |
| `GCP_CLOUDSQL_CONNECTION` | the connection name from step 2, or re-printed here |

The deploy workflow's first step checks for all four with `: "${VAR:?message}"` and
fails immediately (before touching GCP) if any is missing. This prevents a deploy
that silently targets the wrong project.

**Cost:** $0. IAM and WIF have no charge.

## 6. Restore — `infra/06-restore.sh`

The restore script uploads `data/backups/agentic_librarian_FINAL_20260605_014912.sql.gz`
to the GCS bucket, grants the Cloud SQL service agent read access to the bucket, and
runs `gcloud sql import sql` to load it into the database **as the `librarian` user**
(`--user` — the default import user cannot `SET ROLE "librarian"` on PG16, which aborts
the import at the dump's first `OWNER TO librarian` statement; importing as the owner
makes those statements no-ops).

Run this from a clone that actually holds the dump — `data/backups/` is gitignored, so
a fresh clone does NOT have it. If the WSL clone is missing it, copy from the Windows
clone first:

```bash
# From the WSL clone root (adjust the source if your Windows clone lives elsewhere):
mkdir -p data/backups
cp /mnt/c/dev/agentic_librarian/data/backups/agentic_librarian_FINAL_20260605_014912.sql.gz \
   data/backups/
```

### Pre-flight: confirm the vector extension is in the dump

Before running the script, peek at the top of the dump and look for a
`CREATE EXTENSION` line that references `vector`:

```bash
zcat data/backups/agentic_librarian_FINAL_20260605_014912.sql.gz | head -100
```

Scan the output for a line like:
```
CREATE EXTENSION IF NOT EXISTS vector WITH SCHEMA public;
```

**If that line IS present:** proceed directly to running the script — the extension
will be restored from the dump.

**If that line is NOT present:** you must create the extension manually first, because
the dump's `ALTER ... OWNER TO librarian` statements will execute against tables that
depend on the extension, and the import will fail partway through:

```bash
# Reset the postgres superuser password so you can connect
gcloud sql users set-password postgres \
  --instance=librarian-sql \
  --prompt-for-password

# Open an interactive psql session inside Cloud SQL
gcloud sql connect librarian-sql --user=postgres --database=agentic_librarian

# Inside psql:
CREATE EXTENSION IF NOT EXISTS vector;
\q
```

### Run the restore

```bash
bash infra/06-restore.sh
```

The script:
1. Checks that the dump file exists locally and exits with a clear error if not.
2. Copies the dump to `gs://agentic-librarian-prod-backups/`.
3. Looks up the Cloud SQL instance's own service agent email and grants it
   `roles/storage.objectViewer` on the bucket — Cloud SQL imports run as this agent,
   not as you.
4. Runs `gcloud sql import sql` to load the dump. This can take several minutes.

**If the import fails partway through:** the database is in an unknown partial state.
Drop and recreate it before re-running — nothing else depends on this DB until
verification passes:

```bash
gcloud sql databases delete agentic_librarian --instance=librarian-sql
gcloud sql databases create agentic_librarian --instance=librarian-sql
# fix whatever caused the failure, then:
bash infra/06-restore.sh
```

## 7. Verify — `infra/verify_restore.py`

The verification script connects to Cloud SQL and checks six things: row counts for
four tables, that pgvector is installed, that all trope and style embeddings are
populated, and that a vector similarity query returns correctly ordered results.

You cannot connect to Cloud SQL directly from your workstation (no public IP is
exposed). Use the **Cloud SQL Auth Proxy** as a local TCP stand-in.

**Download the proxy** following the instructions at
<https://cloud.google.com/sql/docs/postgres/connect-auth-proxy>. For Linux/WSL:

```bash
curl -o cloud-sql-proxy https://storage.googleapis.com/cloud-sql-connectors/cloud-sql-proxy/v2.11.4/cloud-sql-proxy.linux.amd64
chmod +x cloud-sql-proxy
```

**Set up Application Default Credentials** (one-time — distinct from `gcloud auth
login`, which only authenticates the CLI; the proxy and Google client libraries read
ADC):

```bash
gcloud auth application-default login   # browser flow, like gcloud init
```

(Alternative without ADC: add `--gcloud-auth` to the proxy command to borrow the CLI's
credentials.)

**Start the proxy** (replace `<CONNECTION_NAME>` with the value from step 2 or 5):

```bash
./cloud-sql-proxy <CONNECTION_NAME> --port 5433 &
```

The `&` runs it in the background. The proxy prints `Listening on 127.0.0.1:5433` over
your prompt — press Enter to get the prompt back; it's still running (`jobs` shows it).

**Read the password from Secret Manager** — the secret stores the full DATABASE_URL,
so you can paste it and change only the host:

```bash
gcloud secrets versions access latest --secret=librarian-db-url
# Output looks like:
# postgresql://librarian:<hex-password>@/agentic_librarian?host=/cloudsql/<connection-name>
```

**Run the verification script,** substituting `localhost:5433` for the socket path:

```bash
DATABASE_URL="postgresql://librarian:<hex-password>@localhost:5433/agentic_librarian" \
  python infra/verify_restore.py
```

WSL's bare `python3` has neither pip nor sqlalchemy — route through the app container
instead (the container reaches the WSL-hosted proxy via `host.docker.internal`; no
manual password copying needed):

```bash
URL=$(gcloud secrets versions access latest --secret=librarian-db-url)
DBURL=$(echo "$URL" | sed 's#@/agentic_librarian?host=.*#@host.docker.internal:5433/agentic_librarian#')
docker run --rm -e DATABASE_URL="$DBURL" -v "$PWD":/app -w /app \
  agentic_librarian-app:latest python infra/verify_restore.py
```

All checks must pass before the first deploy. Expected passing output:

```
[OK ] works row count — expected 326, got 326
[OK ] editions row count — expected 335, got 335
[OK ] reading_history row count — expected 331, got 331
[OK ] authors row count — expected 230, got 230
[OK ] pgvector extension installed
[OK ] tropes embeddings populated — 556 rows, 0 NULL embeddings
[OK ] styles embeddings populated — 508 rows, 0 NULL embeddings
[OK ] similarity query returns ordered results — top-3 distances: [...]

All restore checks passed.
```

If any check fails, diagnose and fix before deploying — a broken DB will cause the
live smoke test in the deploy workflow to fail at `/health/db`, rolling back the
service but wasting 10 minutes of CI time.

When done, stop the proxy:

```bash
kill %1   # or: killall cloud-sql-proxy
```

## 8. First deploy

The deploy workflow runs automatically on pushes to `main` that touch the app code,
but for the first deploy you want to trigger it immediately:

**GitHub → Actions → "Deploy to Cloud Run" → Run workflow → (leave branch as `main`) → Run workflow**

The workflow only executes on `main` (`if: github.ref == 'refs/heads/main'`) — if
you trigger it from a feature branch via workflow_dispatch, it will queue and then
cancel. Dispatch from `main`.

**What the workflow does, in order:**

1. **Variables guard** — checks all four `GCP_*` variables are set; fails fast with a
   clear message if any is missing.
2. **Checkout + Python setup** — installs dependencies including the `[dev,claude]`
   extras.
3. **Tests** — runs `pytest -m "not api_dependent and not slow"` against a
   `pgvector/pgvector:pg16` service container spun up inside the runner.
4. **Build** — `docker build -f Dockerfile.api` tags the image with the git SHA.
5. **In-runner smoke test** — starts the image locally (with a dummy `DATABASE_URL`)
   and polls `/health` for up to 30 seconds. A broken image never reaches the
   registry.
6. **Authenticate to GCP** — exchanges the GitHub OIDC token for short-lived GCP
   credentials via Workload Identity Federation (no stored keys involved).
7. **Push image** — pushes to
   `us-central1-docker.pkg.dev/agentic-librarian-prod/librarian/librarian-api:<sha>`.
8. **Deploy to Cloud Run** — deploys `librarian-api` in `us-central1` with:
   `--no-allow-unauthenticated` (IAM-gated), the runtime SA, Cloud SQL attached, the
   `librarian-db-url` secret injected as `DATABASE_URL`, max 1 instance, 512 MB RAM.
9. **Live smoke test** — mints an identity token for the deployer SA and calls
   `<SERVICE_URL>/health` and `<SERVICE_URL>/health/db` with it. If `/health/db`
   returns a non-2xx (which means the DB is unreachable), the workflow exits non-zero
   and the previous revision remains live.

The service URL appears in the output of the "Deploy to Cloud Run" step. The first
run takes roughly 10 minutes.

## 9. Calling the service (it's IAM-gated — a bare browser gets 403)

The service was deployed with `--no-allow-unauthenticated`. Every request must carry
a valid Google identity token for an account that has `roles/run.invoker` on the
service. Your personal `gcloud` login qualifies.

**From the command line:**

```bash
TOKEN=$(gcloud auth print-identity-token)
curl -H "Authorization: Bearer ${TOKEN}" <SERVICE_URL>/works | head -50
```

Replace `<SERVICE_URL>` with the URL from the deploy step output (it looks like
`https://librarian-api-xxxxxxxxxx-uc.a.run.app`).

**Browser-friendly proxy (no token juggling):**

```bash
gcloud run services proxy librarian-api --region us-central1
```

This opens a local port (default 8080) and injects the auth token for you. Open
`http://localhost:8080/works` in any browser. Press Ctrl-C to stop the proxy.

One-time post-restore count check (matches the dump):

```bash
TOKEN=$(gcloud auth print-identity-token)
URL=<SERVICE_URL>
curl -s -H "Authorization: Bearer ${TOKEN}" "${URL}/history" | python3 -c "import json,sys; print('history rows:', len(json.load(sys.stdin)))"   # expect 331
curl -s -H "Authorization: Bearer ${TOKEN}" "${URL}/works?limit=200" | python3 -c "import json,sys; print('works page 1:', len(json.load(sys.stdin)))"   # expect 200
curl -s -H "Authorization: Bearer ${TOKEN}" "${URL}/works?limit=200&offset=200" | python3 -c "import json,sys; print('works page 2:', len(json.load(sys.stdin)))"   # expect 126 (total 326)
```

## 10. Budget — `infra/07-budget.sh`

Sets a $25/month budget on the project with email alerts sent to billing admins at
50%, 90%, and 100% of the budget. Alerts are informational only — GCP never
auto-stops services when a budget threshold is crossed.

```bash
bash infra/07-budget.sh
```

The 50% alert fires at ~$12.50, which is roughly normal spend when Cloud SQL is the
only always-on resource. Think of it as a monthly heartbeat confirming the project is
running normally. The 90% and 100% alerts signal something unexpected — investigate
if you see them.

**Cost:** $0. Budget alerts are free.

## Teardown (if ever needed)

Deleting the project removes every resource — Cloud SQL, Artifact Registry, Secret
Manager, Cloud Run, IAM, WIF — and billing stops within minutes:

```bash
gcloud projects delete agentic-librarian-prod
```

GCP gives you a 30-day grace period before permanently destroying the project. You
can cancel the deletion during that window via the GCP console if you change your
mind.

The catalog's source of truth is the local dev DB and the `data/backups/` dumps in
the WSL clone. Everything in GCP is reproducible by re-running this runbook.
